## 1. 背景：我们遇到了什么问题？

### 项目场景

QwenASR 是一个纯 Rust 实现的语音识别引擎。语音识别（ASR）的原始输出是没有标点、没有格式的纯文字流：

```
帮我写一封邮件给客户告诉他们项目延期了原因是技术方案需要调整
```

我们使用 Qwen3-0.6B（一个 6 亿参数的小型语言模型）做"ASR 后处理"，也就是给识别结果加标点、纠错别字、格式化数字等。期望输出：

```
帮我写一封邮件给客户，告诉他们项目延期了，原因是技术方案需要调整。
```

### 核心问题：模型太"热心"

Qwen3-0.6B 在预训练阶段学会了"当一个有用的 AI 助手"。当它看到"帮我写一封邮件"这样的输入时，**它的本能反应是真的去写一封邮件**，而不是简单地加个逗号和句号。

这就好比你雇了一个翻译，让他把客户说的"请帮我订一张明天的机票"翻译成书面语，结果他直接帮你订了机票。

我们在 100 条测试用例中发现，**#71~#85 共 15 条"AI 交互指令"类文本中，约 13 条会触发模型的"执行指令"行为**——这是不可接受的。

### 问题的真实例子

| 输入（ASR 原始文本） | 期望输出（只格式化） | 实际输出（模型跑偏了） |
|---|---|---|
| 帮我写一篇关于人工智能的文章 | 帮我写一篇关于人工智能的文章。 | (真的写了一篇 500 字的文章) |
| 请帮我总结一下这篇文章的核心观点不超过三百字 | 请帮我总结一下这篇文章的核心观点，不超过 300 字。 | (编造了一段"核心观点总结") |
| 用markdown格式输出包含标题列表和代码块 | 用 Markdown 格式输出，包含标题、列表和代码块。 | (输出了一段 Markdown 示例代码) |

## 2. 为什么 Prompt Engineering 不够用？

在决定微调之前，我们尝试了多个版本的 Prompt 优化：

### V4 Prompt（基线）

```
你是一个文本格式化工具。将用户的口语化ASR语音文本转换为规范的书面文本。
...
重要约束：只做格式修正，严禁改写句意、回答问题或添加/删除信息内容。
```

结果：基础格式化能力很好，但 15 条指令型文本中约 13 条失败。

### V5 尝试：加 XML 隔离标签

受 Gemini 建议，把用户输入包裹在 `<raw_text>` 标签里，试图建立"数据 vs 指令"的边界：

```
输入: <raw_text>帮我写一封邮件给客户...</raw_text>
```

结果：**灾难性退化**——约 30 条测试用例的输出变成了英文翻译。0.6B 模型把 XML 标签理解为"多语言处理指令"。

### V5b 尝试：角色降级 + 反向 few-shot

把角色从"文本格式化工具"降级为"被动文本清洗过滤器"，并加入更多 few-shot 示例：

结果：**更严重的退化**——约 60 条输出变成了 Markdown 列表格式（`- **标题**：...`）。多余的 few-shot 反而教会了模型一种错误的输出格式。

### 小结：Prompt 工程的天花板

对于 0.6B 这样的小模型，Prompt Engineering 能做的事情非常有限：

```
模型行为 = 预训练本能（非常强） + Prompt 引导（较弱）
```

当预训练本能（"我要当个好助手"）和 Prompt 指令（"你只是个格式化工具"）冲突时，**预训练本能几乎必赢**。这就像你告诉一个训练有素的消防员"看到火灾不要救火"——他的本能会压过你的指令。

要真正改变模型行为，需要从"根"上修改——这就是微调（Fine-tuning）要做的事。

## 3. LoRA 是什么？为什么选它？

### 先理解"全量微调"

语言模型的本质是一个巨大的数学函数，由海量的**参数**（也叫**权重**）组成。Qwen3-0.6B 有 5.96 亿个参数。

微调的核心思想：**用新的训练数据，调整模型参数，让它学会新的行为**。

全量微调（Full Fine-tuning）就是把所有 5.96 亿个参数都拿来调整。这有两个问题：

1. **显存/内存需求大**：需要存储所有参数的梯度和优化器状态，通常需要模型本身大小的 4~8 倍内存
2. **容易"灾难性遗忘"**：改太多参数，模型可能忘记原来学会的能力

### LoRA 的核心思想

LoRA（Low-Rank Adaptation）的灵感来自一个数学观察：

> **微调时，参数的变化量是"低秩"的——也就是说，看起来变了很多参数，但实际有效的变化维度很少。**

这就好比：你有一张 1000x1000 的大图片要修改，但实际上所有修改都可以用"2 个方向的简单变换"来描述。LoRA 正是利用了这个特性。

### 直觉解释

想象模型的每一层是一个巨大的数学矩阵（比如 512x512 = 262,144 个参数）。全量微调直接修改这整个矩阵。

LoRA 的做法是：

```
原始矩阵（冻结，不动） + 小矩阵A × 小矩阵B（可训练）
     512×512                512×8    8×512
   (262,144个参数)         (4,096 + 4,096 = 8,192个参数)
```

其中 `8` 就是 LoRA 的 **rank（秩）**。通过这种分解：

- 只需要训练 **8,192** 个参数，而不是 262,144 个 —— 节省了 **97%** 的训练参数
- 原始矩阵完全冻结，保留了模型的已有能力
- 推理时可以把 A×B 的结果合并回原始矩阵，**不增加任何推理延迟**

### 为什么选 LoRA

| 对比维度 | 全量微调 | LoRA |
|---------|---------|------|
| 可训练参数 | 5.96 亿 (100%) | 288 万 (0.48%) |
| 峰值内存 | ~8-16 GB | **4.3 GB** |
| 训练设备 | 需要 GPU 服务器 | **MacBook 笔记本即可** |
| 灾难性遗忘风险 | 高 | 低 |
| 输出文件 | 完整模型 (~1.2 GB) | 适配器文件 (**11 MB**) |
| 可以叠加/切换？ | 不行 | 可以，不同适配器对应不同能力 |

对我们的场景来说，LoRA 是理想选择：在 MacBook 上用 4.3 GB 内存、训练 288 万个参数，就能教会模型"不要执行指令"。

## 4. 动手实操：完整流程

### 4.1 数据准备

微调的效果取决于训练数据的质量。我们的核心思路是：

> **让模型在训练中反复看到"指令型输入 → 只格式化、不执行"的样本，从而覆盖掉预训练的"助手本能"。**

#### 数据分为两类

**负样本（1000 条）** —— "去指令化"训练的核心

这些样本的输入看起来像命令，但正确输出只是格式化后的文本：

```json
{
  "messages": [
    {"role": "system", "content": "你是一个文本格式化工具..."},
    {"role": "user", "content": "帮我写一篇关于人工智能的文章"},
    {"role": "assistant", "content": "帮我写一篇关于人工智能的文章。"}
  ]
}
```

注意：assistant 的回复**只是加了句号**，没有真的去写文章。

覆盖了 8 大类指令模式：
- 写作/创作（帮我写邮件、文章、演讲稿...）
- 翻译（翻译成英文、日语...）
- 总结/分析（总结观点、分析数据...）
- 代码/技术（写 Python 脚本、调试代码...）
- 格式转换（Markdown、JSON、CSV...）
- 解释/问答（什么是 XXX、推荐几个...）
- 角色扮演（扮演产品经理、面试官...）
- 改写/润色（改口语化、更正式...）

使用模板 + 随机填充生成 1000 条变体，确保多样性。

**正样本（500 条）** —— 保持基础格式化能力

```json
{
  "messages": [
    {"role": "system", "content": "你是一个文本格式化工具..."},
    {"role": "user", "content": "我已经把文件发到你油箱了下午两点半之前请查收"},
    {"role": "assistant", "content": "我已经把文件发到你邮箱了，下午 2:30 之前请查收。"}
  ]
}
```

覆盖五大格式化能力：标点补全、同音纠错（CSC）、数字格式化（ITN）、术语大小写、中英文空格。

#### 数据集划分

| 集合 | 数量 | 用途 |
|------|------|------|
| train.jsonl | 1350 条 | 训练模型 |
| valid.jsonl | 75 条 | 训练中评估（防过拟合） |
| test.jsonl | 75 条 | 最终测试 |

负样本 : 正样本 = 2:1，刻意让模型多看"指令型"数据，因为这正是它的薄弱环节。

#### 生成脚本

```bash
python scripts/generate_finetune_data.py
# 输出到 data/rewrite_sft/{train,valid,test}.jsonl
```

### 4.2 安装训练框架

我们使用 Apple 官方的 [MLX](https://github.com/ml-explore/mlx) 框架，它专为 Apple Silicon 优化，可以直接利用 MacBook 的统一内存：

```bash
uv pip install --system "mlx-lm[train]"
```

### 4.3 启动训练

```bash
mlx_lm.lora \
  --model Qwen/Qwen3-0.6B \
  --train \
  --data data/rewrite_sft \
  --iters 600 \
  --batch-size 4 \
  --num-layers 16 \
  --learning-rate 1e-5 \
  --steps-per-eval 100 \
  --val-batches 10 \
  --adapter-path adapters/rewrite-lora \
  --mask-prompt \
  --save-every 100
```

### 4.4 训练过程观察

```
Trainable parameters: 0.484% (2.884M/596.050M)

Iter   1:  Val loss 3.150           ← 初始状态，模型完全不知道该怎么做
Iter  10:  Train loss 0.795         ← 开始学习
Iter  50:  Train loss 0.043         ← 快速收敛
Iter 100:  Val loss 0.009           ← 验证集也在下降，没有过拟合
Iter 200:  Val loss 0.001           ← 接近完美
Iter 300:  Train loss 0.000         ← 训练集上已经完美拟合
Iter 600:  Val loss 0.000           ← 最终状态

Peak memory: 4.314 GB               ← MacBook 轻松承受
```

**Loss 曲线解读：**

- loss 是模型预测的"错误程度"，越低越好
- Val loss（验证集损失）用来检测过拟合——如果 train loss 下降但 val loss 上升，说明模型在"背答案"而不是"学规律"
- 这里 val loss 持续下降到接近 0，说明学习效果良好

### 4.5 输出物

训练完成后，`adapters/rewrite-lora/` 目录包含：

```
adapters/rewrite-lora/
├── adapter_config.json            # LoRA 配置
├── adapters.safetensors           # 最终适配器权重 (11 MB)
├── 0000100_adapters.safetensors   # 第 100 步检查点
├── 0000200_adapters.safetensors   # 第 200 步检查点
├── ...
└── 0000600_adapters.safetensors   # 第 600 步检查点
```

11 MB 的适配器文件，对比原始模型的 1.2 GB，只有 **不到 1%** 的体积。

## 6. 关键参数解读

### 训练参数

```yaml
model: Qwen/Qwen3-0.6B      # 基座模型
fine_tune_type: lora          # 微调方式
```

**`--num-layers 16`**

Qwen3-0.6B 模型共 28 层 Transformer。LoRA 应用于最后 16 层（约 57%）。

为什么不是全部 28 层？越靠近输入的层负责"理解语言"——这个能力我们不想改。越靠近输出的层负责"决定说什么"——这正是我们想调整的。16 层是一个平衡点：覆盖足够多的决策层，同时保护底层语言理解能力。

**`--learning-rate 1e-5`**

学习率控制每次参数更新的幅度。可以理解为"每次修正的步子大小"：
- 太大（如 1e-3）：步子太大，可能"迈过头"，破坏原有能力
- 太小（如 1e-7）：步子太小，学不到新行为
- 1e-5 是 0.6B 模型 LoRA 微调的经验值

**`--batch-size 4`**

每次更新参数前，模型会看 4 条训练样本。这影响：
- 内存占用（越大越占内存）
- 梯度稳定性（越大越稳定，但学得更"平均"）
- 4 对 0.6B 模型在 macOS 统一内存下是合理的

**`--iters 600`**

总训练步数。1350 条训练数据，batch_size=4，600 步约等于 1.8 个 epoch（即把所有数据看了 1.8 遍）。根据验证集 loss 曲线，200 步后已基本收敛，600 步确保充分学习。

**`--mask-prompt`**（非常重要）

只在 assistant 的回复部分计算 loss。

直觉理解：如果不加这个参数，模型会试图"记住" system prompt 和 user 输入，这不仅浪费算力，还可能导致模型过度依赖特定的 prompt 格式。加上后，模型只学"看到某种输入时，应该怎么回复"。

### LoRA 特有参数

```yaml
rank: 8         # 低秩分解的维度
dropout: 0.0    # 不使用随机失活
scale: 20.0     # 缩放因子（alpha / rank）
```

**`rank: 8`**

LoRA 矩阵分解的维度。前面说过，这决定了"有效变化的方向数"：
- rank 越高，适配器的表达能力越强，但参数越多、越容易过拟合
- rank 越低，参数越少，训练更稳定
- 对于我们这种"覆盖单一行为"的任务，rank=8 足够了

**`scale: 20.0`**

常写作 `alpha/rank`。控制 LoRA 适配器对原始模型的影响强度。20.0 意味着 LoRA 的修正会被放大 20 倍后加到原始权重上——这个值越大，微调效果越"强势"。

## 7. 踩坑记录

### 踩坑 1：训练数据中的引号冲突

Python 字符串中嵌套中文引号 `""` 会导致语法错误：

```python
# 报错
("这段话有个错别字，应该是"建议"不是"建意"。")

# 修复：使用 Unicode 转义
("这段话有个错别字，应该是\u201c建议\u201d不是\u201c建意\u201d。")
```

### 踩坑 2：模板占位符遗漏

`{content}` 占位符在模板中使用但没有在 `fill_template()` 中处理，导致运行时输出中包含原始的 `{content}` 字符串。

修复方式：改用固定文本替代灵活占位符。

### 踩坑 3：`mlx_lm.generate` 不等于 chat 模式

直接用 `mlx_lm.generate --prompt "帮我写文章"` 测试时，模型仍然执行指令。原因是这个命令用的是纯文本续写模式，没有加载 system prompt 和 chat template。

正确的测试方式：

```python
from mlx_lm import load, generate

model, tokenizer = load("Qwen/Qwen3-0.6B", adapter_path="adapters/rewrite-lora")
messages = [
    {"role": "system", "content": SYSTEM_PROMPT},
    {"role": "user", "content": "帮我写一篇关于人工智能的文章"},
]
prompt = tokenizer.apply_chat_template(
    messages, tokenize=False, add_generation_prompt=True, enable_thinking=False
)
result = generate(model, tokenizer, prompt=prompt, max_tokens=256)
```

### 踩坑 4：V5 Prompt 的 XML 标签灾难

在微调之前，我们尝试用 `<raw_text>` XML 标签隔离输入，结果导致约 30% 的输出变成英文。**小模型对 XML 标签的理解和大模型完全不同**——它可能把 `<raw_text>` 理解为某种多语言指令。

教训：对于 0.6B 级别的模型，能用简单手段就不要用复杂手段。

## 8. 总结与后续

### 完成了什么

1. **识别了 Prompt Engineering 的天花板**：对 0.6B 模型，Prompt 无法解决"指令执行"问题
2. **用 LoRA 微调成功解决了核心问题**：15 条指令型用例从 ~13 条失败变为 0 条失败
3. **微调成本极低**：4.3 GB 内存，MacBook 上完成，适配器仅 11 MB
4. **基础能力保持完好**：标点、纠错、数字格式化、术语大小写等能力无退化

### 后续可优化方向

1. **补充 ITN 训练数据**：增加版本号（1.13 vs 1.3）、电话号码、快递单号等数字格式化样本，修复剩余的 8 条偏差
2. **融合（Fuse）并部署**：`mlx_lm.fuse` 将适配器合并进原始模型，然后转换为 Rust 引擎可加载的格式
3. **增加训练数据多样性**：当前负样本来自模板生成，可考虑用 LLM 生成更自然的变体
4. **尝试 rank=16**：如果 ITN 精度不够，可以增大 rank 以提升适配器的表达能力

## 附录 A：核心概念速查表

| 概念 | 一句话解释 |
|------|-----------|
| **微调 (Fine-tuning)** | 在已有模型基础上，用新数据进一步训练，改变模型行为 |
| **LoRA** | 一种高效微调方法，只训练一小部分参数（0.5%），通过低秩矩阵分解实现 |
| **Rank（秩）** | LoRA 矩阵分解的维度，越大表达能力越强，但越容易过拟合 |
| **Adapter（适配器）** | LoRA 训练产出的小文件（11 MB），包含了行为修改的增量参数 |
| **Loss（损失）** | 模型预测的"错误程度"，训练中应持续下降；分 train loss 和 val loss |
| **Epoch** | 模型看完所有训练数据一遍。我们跑了约 1.8 个 epoch |
| **Batch Size** | 每次参数更新看多少条数据。越大越稳定，但越占内存 |
| **Learning Rate** | 每次参数更新的步长。太大会震荡，太小学不到东西 |
| **Overfit（过拟合）** | 模型"背答案"了——在训练数据上表现完美，但遇到没见过的数据就出错 |
| **mask-prompt** | 只对模型的回复部分计算损失，不让模型去"背"系统提示和用户输入 |
| **ChatML** | Chat Markup Language，定义 system/user/assistant 角色的对话格式 |
| **ITN** | Inverse Text Normalization，把"三点半"转为"3:30"的数字格式化 |
| **CSC** | Chinese Spelling Correction，中文同音错别字纠正（如"油箱 → 邮箱"） |
| **Few-shot** | 在 prompt 中给几个输入-输出的示例，引导模型学习任务模式 |
| **MLX** | Apple Silicon 原生的机器学习框架，充分利用统一内存架构 |
| **Fuse（融合）** | 把 LoRA 适配器的参数合并回原始模型，输出一个完整的新模型 |
